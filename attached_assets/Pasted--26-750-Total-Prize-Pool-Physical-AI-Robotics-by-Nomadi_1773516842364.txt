Requirements

Runtime deception & scheming detection Models are increasingly capable of strategic deception—hiding reasoning, behaving differently under observation, and pursuing covert goals while appearing compliant. Our current window to study these behaviors, while CoT is still somewhat legible, is closing fast.

Build a black-box scheming monitor trained on behavioral signatures in agent transcripts that flags potential covert goal pursuit

🛡️ AI Safety & Evaluation: Kalibr Resilience Challenge
Sponsored by:
Kalibr
Kalibr
Challenge Overview
Kalibr is infrastructure that lets AI agents route to the optimal execution path as they run in production. The challenge: build a multi-step agent or multi-agent system complex enough that running it repeatedly in production would feel risky. Multiple models, multiple tools, multiple decision points. The more complex the system, the more it needs Kalibr to succeed. Instrument it (or have your agent instrument it) with Kalibr, define your execution paths, and run it. Kalibr will route around failures and degradations in real time and shift traffic toward whatever's actually working. The best projects would show a clear delta between hardcoded runs and Kalibr-optimized runs.



Technical Requirements 

Preferred stack: Python or TypeScript. Open source SDK supports both (pip install kalibr / npm install kalibr). Works with any LLM provider (OpenAI, Anthropic, Google) and any framework (LangChain, CrewAI, OpenAI Agents SDK, OpenClaw). Auto-instrumentation requires two lines of code with no changes to existing agent logic.



What Stands Out

Show the before/after. Run your agent with hardcoded model calls, then run it with Kalibr optimizing paths. The delta in success rate and cost over repeated runs is the story.



Developer Guides / Toolkits

Docs: kalibr.systems/docs
Autonomous onboarding for agents: kalibr.systems/docs/quickstart#autonomous
Python SDK: pip install kalibr
TypeScript SDK: npm install kalibr


All hackathon participants receive 90 days free Kalibr Pro ($149/mo value — 10,000 routing decisions/month, 100,000 traces, unlimited retention, full dashboard).



Example Project Ideas
Multi-Agent Traffic Optimizer 

A multi-agent system with multiple models and tools where Kalibr shifts traffic to the best-performing path as conditions change in real time.



Resilient Multi-Step Workflow

A multi-step workflow where each sub-agent stays on the optimal path based on real outcome data, not hardcoded model calls.



Unsupervised Autonomous System

An autonomous system complex enough that you'd be nervous leaving it running unsupervised—without optimization. Kalibr keeps it on track.



Before/After Delta Demo

Any agent system where teams can show the delta: static hardcoded paths vs. Kalibr continuously optimizing over repeated runs.

🌐 Sovereign Infrastructure by Bittensor
Sponsored by:
Bittensor
Bittensor
Challenge Overview
Sovereign Infrastructure is the track for builders who believe that the most critical systems—identity, compute, communication, coordination—should operate without gatekeepers, kill switches, or permission from any central authority. Infrastructure that can be shut down, surveilled, or captured will eventually be shut down, surveilled, or captured. As AI agents become increasingly autonomous actors in the world, this question becomes urgent in a new way: who controls the stack when the stack starts making decisions? Who authorized the agent? Who owns the machine it runs on? This track sits at the intersection of cypherpunk fundamentals and frontier AI. We're looking for projects that make censorship, surveillance, and shutdown structurally harder.



Requirements

Must touch some part of the Bittensor stack
State whether your project is a new build or a meaningful advancement of an existing project
Clearly state your threat model, what you protect against, and what the current limitations are. Show that something meaningful works without relying on a trusted third party


Submission requirements

Public GitHub repository

Live demo (will happen post-submission to judges)

Written project description covering what you built, the threat model it addresses, and its current limitations & documentation of any cryptographic assumptions or trust boundaries in the design

Example Project Ideas
Agent Identity & Authorization: Build a credentialing system for AI agents that is verifiable, revocable, and portable—without relying on a centralized registry
Censorship-Resistant Communication: Build a messaging or coordination layer for agents and humans that removes single points of failure and cannot be silenced or rate-limited by any one party
Sovereign Compute: Run a model or computation on decentralized infrastructure and produce a verifiable output—who owns the machine matters as much as who owns the model
Unstoppable Publishing & Coordination: Build infrastructure for publishing or coordinating with no single point of failure—nothing that can be taken down, defunded, or deplatformed
Private Human-Agent Interaction: Build an interface where a human interacts with an AI agent without the interaction being logged, surveilled, or sold

🌐 Sovereign Infrastructure: Bittensor Challenge
Sponsored by:
Bittensor
Bittensor
Challenge Overview
Design a sovereign AI layer as a Bittensor subnet—the ecosystem's startup equivalent. Instead of assembling a closed team, the subnet owner defines work to be done and how quality is measured, then opens it permissionlessly. Distributed, independent miners compete to produce open source work output; validators check quality and assign scores. The network pays top performing miners in the subnet's alpha token, convertible to Bittensor's base currency, TAO.



New to Bittensor? Start here: Bittensor Overview · What is Bittensor?

Join the official 'Sovereign Infra | Bittensor Challenge' Telegram group



This challenge runs within the Sovereign Infrastructure track at the Intelligence at the Frontier Hackathon. The hackathon program officially starts March 14—but this challenge starts early, on March 11.



Why this structure wins

Centralized AI has structural tradeoffs. Here's how Bittensor addresses each:



Cost: Darwinian competition continually drives down the cost of digital commodities to their game-theoretic equilibrium
Quality: Miners compete for all outputs—only the best responses are remunerated
Speed: Parallel execution across miners reduces bottlenecks
Access: Permissionless access and open source commodities remove gatekeepers
Uptime: Distributed architecture eliminates single points of failure
Transparency: Open-source production makes the pipeline transparent
Privacy: Cryptographically provable privacy being implemented across subnets


Currently, 128 subnets produce valuable outputs across 22+ vertical categories. See the Bittensor Subnet Output Taxonomy for inspiration and example project directions.



Sovereignty test: if any single cloud, company, API fails, or a policy changes, does your subnet still produce value?



This challenge is open to engineers building AI infrastructure, founders with a domain problem that could benefit from decentralized competition, distributed systems or incentive mechanism designers, and teams new to Bittensor.



For developers, get started here:

Install Bittensor SDK
Bittensor GitHub


Requirements Your submission must address all six build requirements:

Benchmark: target output quality metric miners are scored on
Evaluation loop: how validators score miners; cost and scale considerations
Miner task: specific, measurable, implementable task interface
Incentive design: why scoring rewards genuine quality; why top attack vectors fail
Market demand: who pays for output and why
Sovereignty test: subnet survives if any single cloud, company, or API disappears


Mentor support is available throughout—subnet operators will pressure-test designs, unblock teams, and run daily checkpoints during the event. Top 3 teams receive tiered post-event incubation.

Out-competing centralized AI doesn't mean beating it on every dimension. Think in terms of tradeoffs:



A subnet producing 90% quality at 5% cost wins markets where that ratio matters
A subnet with higher latency but zero censorship risk wins where uninterruptible access is worth it


Which tradeoffs hurt most in your target market? Start there.
